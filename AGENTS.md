# AGENTS.md

本文件是 AgentKit 的项目级 Codex 指令。处理本仓库任务时，优先遵循这里的
约定；如果子目录后续提供更具体的 `AGENTS.md` 或 `AGENTS.override.md`，以更
靠近当前工作目录的说明为准。

## 项目定位

- AgentKit 是面向 AI 智能体的工具 Kit 合集，不是单一大型应用。
- 工具主要供智能体调用，因此接口必须稳定、可脚本化、可自动化验证。
- 默认使用 Python 实现工具。
- 默认使用 `uv` 管理依赖、虚拟环境、脚本和测试命令。
- 除非有明确记录的例外，每个工具都应同时支持 CLI 和 MCP 两种使用方式。

## 架构原则

- 每个新工具必须拥有独立的仓库根目录，例如 `x-com/`。工具目录承载该工具的
  设计、文档、源码入口和测试说明；Python 包内部命名可使用下划线形式，例如
  `x_com`。
- 将核心逻辑与运行入口分离。CLI 和 MCP 只是适配层，不应承载主要业务逻辑。
- 同一个工具的 CLI 与 MCP 入口必须复用同一套核心服务、输入模型和输出模型。
- 核心逻辑应保持可测试，不依赖 CLI 框架、MCP server 对象、终端格式化、
  `sys.exit`、交互式输入或隐式全局状态。
- 对外边界优先使用结构化、JSON 兼容的数据模型。
- 错误应在核心层表达为明确的领域错误，再由 CLI/MCP 适配层映射成各自的
  错误响应。

推荐工具结构：

```text
<tool-name>/
  README.md
  pyproject.toml
  src/<tool_package>/
    core.py        # 核心逻辑或服务
    models.py      # 输入、输出、配置、错误模型
    cli.py         # CLI 参数解析、输出格式、退出码
    mcp.py         # MCP tool/resource 注册与协议映射
    __init__.py
  tests/
    test_core.py
    test_cli.py
    test_mcp.py
```

如果现有代码已经形成了不同但一致的结构，优先延续现有结构。

## CLI 约定

- 每个工具默认提供 CLI 入口，便于 shell、脚本和本地调试使用。
- CLI 默认应支持非交互式运行；不要把交互式提示作为唯一输入方式。
- 提供清晰的 `--help`。
- 为智能体和自动化场景提供 `--json` 或等价结构化输出。
- 人类可读输出可以存在，但不能替代结构化输出。
- 失败时输出可读错误到 stderr，并返回有意义的非零退出码。

## MCP 约定

- 每个工具默认提供 MCP 入口，便于智能体通过协议调用。
- MCP tool 名称应稳定、描述清晰，避免随意重命名。
- 明确输入 schema、输出 schema、错误模式和副作用。
- MCP 响应不得混入 CLI 专用排版。
- 长时间操作应考虑超时、取消或进度反馈。
- 密钥、token 和本地凭据必须通过调用方配置传入，不得写入仓库。

## Python 与 uv 约定

- 使用 `pyproject.toml` 管理包信息、依赖、脚本和工具配置。
- 使用 `uv add`、`uv remove`、`uv sync`、`uv run` 管理依赖和命令。
- 新增依赖前先确认标准库或已有依赖是否足够。
- 仅在依赖能显著降低复杂度、提供可靠协议实现或减少维护风险时引入。
- 优先使用类型标注。边界模型可使用 dataclass、Pydantic 或项目已有 schema
  方案。
- 保持模块职责清晰，避免把核心逻辑、CLI 解析和 MCP 注册堆在同一个文件中。

## 开发流程

修改前：

- 先检查现有目录结构、命名、测试方式和依赖管理方式。
- 明确改动影响核心逻辑、CLI、MCP、schema、文档中的哪些部分。
- 若涉及公共行为变化，优先补充或更新测试。

实现时：

- 优先改核心层，再连接 CLI/MCP 适配层。
- 保持适配层轻量，避免重复实现业务逻辑。
- 在输入边界做校验，在输出边界保持结构稳定。
- 避免隐藏网络调用、隐式文件系统写入和不可预测副作用。
- 保留用户已有改动，不回滚无关文件。

## 验证要求

优先使用项目定义的本地命令。如果还没有更具体的脚本，默认验证顺序为：

```bash
uv run pytest
uv run ruff check
uv run ruff format --check
```

修改单个工具时，至少验证对应核心测试；如果 CLI 或 MCP 契约变化，也要覆盖
对应入口的冒烟测试。

## 文档要求

新增或修改工具时，相关文档应说明：

- 工具用途，以及智能体应在什么场景使用。
- CLI 用法和示例。
- MCP tool 名称、输入 schema、输出 schema。
- 必需配置、环境变量和凭据来源。
- 重要限制、副作用和失败模式。

文档应尽量靠近对应工具；只有跨工具规范或总览才放到仓库顶层。

## 安全与质量

- 不提交 secret、token、真实 MCP 凭据或本地私有配置。
- 文件系统写入必须目标明确，避免宽泛扫描后批量写入。
- 对外部命令、网络调用和删除操作保持保守。
- 生成产物只有在被明确视为源资产时才纳入版本控制。
- 优先选择简单、可检查、可测试的实现。
